MAIN PLAN: Operating-Pack Platform для AI-агентов

1. Определение продукта
Operating-Pack Platform — это система, которая из фактов репозитория и ответов команды собирает и постоянно поддерживает актуальный operating-pack для AI-агентов, чтобы агент работал корректно с первого сообщения и не терял контекст со временем.

2. Проблема
- AI-агент стартует «вслепую»: не знает реальную архитектуру, правила и границы проекта.
- Даже при наличии документации она быстро устаревает и расходится с кодом.
- Контекст между чатами теряется: handoff и состояние задач не стандартизированы.
- Поведение агента нестабильно: нет формализованных правил, когда обновлять контекст и как фиксировать прогресс.
- Команда тратит время на повторное объяснение базы вместо выполнения задач.

3. Основной цикл продукта
1) Пользователь запускает инструмент на репозитории.
2) Система строит фактическую модель проекта (скан).
3) Система задает адаптивные вопросы по невыводимым вещам (правила, границы, workflow).
4) Генерируется operating-pack (человеко-читаемые + машинно-читаемые артефакты).
5) Выполняется валидация полноты и согласованности.
6) Пользователь принимает пакет как baseline.
7) В режиме сопровождения система отслеживает изменения репозитория.
8) Система выявляет устаревание operating-pack и формирует предложения обновлений.
9) При нехватке данных запускаются микро-опросы.
10) Обновления подтверждаются и становятся новой версией operating-pack.
11) Цикл повторяется непрерывно.

4. Карта стадий

Стадия 1: Сканирование репозитория
- Цель: получить фактическую модель проекта, а не предположения.
- Вход: структура файлов, манифесты зависимостей, конфиги CI/CD, скрипты, тесты, docker/infra файлы, существующая документация.
- Выход: Project Fact Model (модули, точки входа, окружения, команды, зоны неопределенности, confidence score).
- Проблемы: шум от сгенерированных файлов, частичная поддержка стеков, ложные выводы в монорепо.
- Улучшения: стек-адаптеры, фильтры шума, confidence-порог с обязательными уточнениями, better heuristics по типам проектов.

Стадия 2: Адаптивный опросник
- Цель: собрать то, что нельзя надежно вывести из кода (политики, границы, handoff-практики).
- Вход: Project Fact Model + список неопределенностей + шаблоны ролей.
- Выход: Policy/Workflow Model (правила поведения агента, эскалации, формат handoff, правила обновления контекста, критерии done).
- Проблемы: усталость от вопросов, неоднозначные ответы, противоречия.
- Улучшения: ветвящийся опрос, лимит вопросов, предзаполненные варианты из контекста скана, встроенная проверка конфликтов ответов.

Стадия 3: Генерация operating-pack
- Цель: превратить факты + правила в операционную систему для агента.
- Вход: Fact Model + Policy Model + шаблоны operating-pack.
- Выход: согласованный пакет файлов (инструкции старта, архитектура, состояние, handoff, поведенческие правила, трекинг задач, машинный манифест).
- Проблемы: «красивая, но пустая» документация, пересечение секций, лишняя многословность.
- Улучшения: доменные шаблоны, контроль плотности/конкретики, role-specific варианты пакета.

Стадия 4: Валидация
- Цель: доказать, что пакет полный, непротиворечивый и применимый.
- Вход: сгенерированный operating-pack + Fact/Policy Model.
- Выход: Validation Report (критические пробелы, конфликты, несоответствия, рекомендации доработки).
- Проблемы: слабая проверка реальной применимости, ложные срабатывания.
- Улучшения: многоуровневая валидация (структурная, семантическая, сценарная), калибровка правил по реальным сессиям агентов.

Стадия 5: Поддержка и обнаружение устаревания
- Цель: поддерживать соответствие evolving repo и agent operating layer.
- Вход: git-изменения, изменения зависимостей, CI-события, история версий operating-pack.
- Выход: Drift Events, предложения патчей operating-pack, микро-опросы, новая версия пакета.
- Проблемы: много шума, пропуск критичных изменений, «алерт-усталость».
- Улучшения: impact scoring, приоритизация по риску, digest-режимы, автообновления только для low-risk зон.

5. Объем V1 (Builder)

Что входит в V1 (MVP-обязательно)
- Одноразовый запуск на одном репозитории.
- Базовый скан ключевых артефактов проекта.
- Адаптивный опросник с ограниченным числом вопросов.
- Генерация operating-pack по фиксированной схеме.
- Базовая валидация полноты и консистентности.
- Ручной перезапуск генерации.

Состав operating-pack в V1
- PROJECT_ARCHITECTURE — карта модулей, точек входа, ключевых зависимостей.
- PROJECT_STATE — текущее состояние проекта, ограничения, известные риски.
- FIRST_MESSAGE_INSTRUCTIONS — как агенту стартовать работу с первого сообщения.
- HANDOFF_PROTOCOL — как передавать контекст между чатами.
- AGENT_BEHAVIOR_RULES — границы, стиль действий, эскалации.
- CONTEXT_UPDATE_POLICY — когда и как обновлять контекст.
- TASK_TRACKING_PROTOCOL — единый формат фиксации прогресса задач.
- OPERATING_PACK_MANIFEST — машинно-читаемая версия, источники, confidence.
- VALIDATION_REPORT — результаты проверок и пробелы.

Что бесплатно (Free Builder)
- Полный V1-пайплайн: scan -> questionnaire -> generation -> validation.
- Локальный экспорт operating-pack.
- Ручная регенерация по запросу.

Критерии успеха V1
- Время до первого пригодного operating-pack: до 20-30 минут.
- Не менее 85% обязательных секций заполнены без ручного дописывания.
- В первом рабочем чате агент начинает полезные действия с минимумом уточнений.
- Критические ошибки валидации: менее 10% запусков.

6. Объем V2 (Maintainer, SaaS)

Что входит в V2
- Непрерывный мониторинг изменений репозитория.
- Детекция устаревших секций operating-pack.
- Предложения обновлений (diff-патчи к секциям).
- Микро-опросы при нехватке данных.
- История версий operating-pack и changelog причин изменений.
- Дашборд «свежести контекста» и риск устаревания.

Что в подписке (SaaS)
- Все функции continuous maintenance.
- Автоматические уведомления о drift-событиях.
- Приоритизация обновлений по влиянию на работу агента.
- Командная работа с ревью/аппрувом обновлений operating-pack.

Критерии успеха V2
- Доля релевантно обнаруженных drift-событий: не менее 80%.
- Ложные алерты: не более 20-25%.
- Время от изменения репо до предложения апдейта: до 24 часов.
- Снижение инцидентов из-за устаревшего контекста: не менее 50%.

7. Объем V3 (Governance Layer)

Что входит в V3
- Политики управления operating-pack на уровне команды/организации.
- Роли и зоны ответственности за секции operating-pack.
- Approval workflows для критичных изменений правил агента.
- Аудит-трейл: кто/почему/когда менял operating layer.
- Политики обязательной валидации перед применением обновлений.
- Governance-метрики качества операционного слоя по проектам.

Критерии успеха V3
- Не менее 90% изменений operating-pack имеют owner + reason + approval trail.
- Снижение нарушений агентом организационных правил: не менее 60%.
- Подготовка аудиторского отчета по изменениям operating-pack: минуты, не часы.

8. Что исключено из MVP
- Полноценная автогенерация PR и автослияние изменений без человека.
- Сложная multi-repo федерация с кросс-репо зависимостями.
- Глубокая персонализация под много ролей агентов в одном проекте.
- Расширенная аналитика продуктивности команды.
- «Умные» автокоррекции поведения агента в рантайме.
- Enterprise governance-функции (RBAC, комплаенс-пакеты, SSO-политики).

9. Риски
- Плохое сканирование репозитория. Митигировать confidence score, explicit unknowns и обязательные уточняющие вопросы.
- Плохая логика опросника. Митигировать адаптивными ветками, лимитом вопросов и валидацией противоречий.
- Слабая валидация. Митигировать трехслойной проверкой: структура, смысл, сценарная применимость.
- Проблемы актуальности. Митигировать приоритизацией drift по риску и микро-опросами только по критичным пробелам.
- Излишняя сложность. Митигировать жестким scope-control: сначала надежный V1 контур, потом автоматизация.

10. План улучшений после MVP
- Ввести telemetry-петлю качества: где пользователи вручную переписывают пакет и почему.
- Собрать эталонный набор репозиториев и регулярно прогонять регрессии качества генерации.
- Улучшать сначала самые «дорогие» ошибки: неверная архитектура, неверные first-message инструкции, сломанный handoff.
- Перейти от статической к сценарной валидации на типовых агентных задачах.
- Добавить обучение приоритизации drift-событий по факту принятия/отклонения обновлений.
- Постепенно вводить governance, когда стабильность V2 подтверждена метриками.

11. Рекомендуемый порядок реализации
1) Зафиксировать каноническую схему operating-pack (обязательные секции и machine-readable manifest).
2) Реализовать Stage 1 скан и модель фактов с confidence/unknowns.
3) Реализовать Stage 2 адаптивный опросник на основе unknowns.
4) Реализовать Stage 3 генератор operating-pack по шаблонам V1.
5) Реализовать Stage 4 валидатор и понятный отчет о пробелах.
6) Провести пилот на реальных репозиториях и скорректировать шаблоны/вопросы.
7) Запустить V1 как стабильный Free Builder.
8) Добавить события изменений и детектор устаревания (ядро V2).
9) Добавить предложения обновлений и микро-опросы.
10) Запустить SaaS слой V2 с метриками качества и шумоподавлением.
11) Добавить governance-процессы V3 после стабилизации точности V2.
