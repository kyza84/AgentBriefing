V1 APPROVAL PLAN
Operating-Pack Platform — План выполнения и согласования V1 Builder

0) Назначение документа
Этот документ фиксирует практический план реализации V1 Builder для согласования между product/owner и исполнением.
Цель V1: стабильно генерировать рабочий operating-pack для одного репозитория за один запуск.

1) Цель V1 и границы

1.1 Цель V1 (что должно быть достигнуто)
- Получать из репозитория и ответов пользователя операционно пригодный operating-pack.
- Обеспечить старт AI-агента с первого сообщения без длительной ручной подводки.
- Дать пользователю отчет о полноте/пробелах и понятный путь доработки.

1.2 In Scope (обязательно в V1)
- Сканирование репозитория и извлечение фактов.
- Адаптивный опросник по невыводимым из кода вопросам.
- Генерация operating-pack по фиксированной схеме.
- Валидация полноты/согласованности.
- Локальный экспорт пакета и ручной перезапуск генерации.

1.3 Out of Scope (исключено из V1)
- Непрерывный мониторинг репо и автоподдержка (это V2).
- Автоматические PR/auto-merge по обновлениям operating-pack.
- Multi-repo и организационный governance (это V3).
- Расширенная аналитика и enterprise-политики.

2) Целевой результат V1 (Definition of Done)

V1 считается выполненным, если одновременно соблюдены условия:
- На пилотных репозиториях формируется полный operating-pack V1-структуры.
- Валидация выявляет пробелы и классифицирует их по критичности.
- Пользователь может пройти цикл scan -> questionnaire -> generation -> validation без ручных обходов.
- Время цикла в норме: до 20-30 минут для репозитория среднего размера.
- Качество: не менее 85% обязательных секций заполняются без ручного редактирования.

3) Формат V1 operating-pack (фиксируем как контракт)

Обязательные артефакты:
- PROJECT_ARCHITECTURE
- PROJECT_STATE
- FIRST_MESSAGE_INSTRUCTIONS
- HANDOFF_PROTOCOL
- AGENT_BEHAVIOR_RULES
- CONTEXT_UPDATE_POLICY
- TASK_TRACKING_PROTOCOL
- OPERATING_PACK_MANIFEST (машинно-читаемый)
- VALIDATION_REPORT

Правило согласования:
- Любая секция либо заполнена фактами/правилами, либо содержит явный блок UNKNOWN + причина + как закрыть пробел.

4) План выполнения V1 по этапам

Этап A. Product Contract Freeze (заморозка контракта V1)
Цель:
- Зафиксировать единый продуктовый контракт, чтобы не расползался scope.
Вход:
- MAIN_PLAN.txt + требования owner.
Выход:
- Утвержденный V1-контракт: scope, формат пакета, метрики успеха, список NFR.
Критерии приемки:
- Подписаны In Scope/Out of Scope.
- Подписан список обязательных артефактов operating-pack.
Основные риски:
- Попытка добавить V2-функции в V1.
Контроль:
- Change policy: новые идеи логируются в backlog V2/V3, но не входят в текущий релиз.

Этап B. Репозиторный Scanner Baseline
Цель:
- Стабильно извлекать факт-модель проекта с уровнем уверенности.
Вход:
- Реальный репозиторий пользователя.
Выход:
- Project Fact Model: структура модулей, входные точки, окружения, зависимости, команды/скрипты, unknowns, confidence.
Критерии приемки:
- Скан выделяет ключевые компоненты проекта и отмечает неизвестные зоны.
- В отчете скана есть confidence и список unresolved unknowns.
Риски:
- Шум (generated/vendor файлы), ложные выводы.
Контроль:
- Фильтры шума + whitelist/blacklist типов файлов.
- Явная маркировка предположений как hypothesis, не как факт.

Этап C. Адаптивный Questionnaire V1
Цель:
- Дособрать правила и границы, которых нет в коде.
Вход:
- Unknowns из сканера + базовые шаблоны вопросов.
Выход:
- Policy/Workflow Model: правила агента, эскалации, handoff-практики, обновление контекста, criteria of done.
Критерии приемки:
- Вопросы задаются только там, где не хватает уверенности/фактов.
- Есть детектор противоречивых ответов и запрос уточнения.
- Общее число вопросов ограничено (анти-усталость).
Риски:
- Слишком длинный опрос, неоднозначные ответы.
Контроль:
- Приоритизация вопросов по критичности.
- Короткие формулировки и предзаполненные варианты.

Этап D. Generator V1 (сборка operating-pack)
Цель:
- Сформировать операционный пакет для практической работы агента.
Вход:
- Fact Model + Policy/Workflow Model + шаблон V1.
Выход:
- Полный operating-pack из 9 обязательных артефактов.
Критерии приемки:
- Нет пустых обязательных секций без UNKNOWN-блока.
- Секции не конфликтуют между собой по ключевым правилам.
- FIRST_MESSAGE_INSTRUCTIONS и HANDOFF_PROTOCOL содержат исполнимые шаги.
Риски:
- «Красивая» документация без операционной полезности.
Контроль:
- Правило операционности: каждая секция должна содержать действия, условия, ограничения.

Этап E. Validator V1
Цель:
- Проверить полноту, согласованность и применимость пакета.
Вход:
- Сгенерированный operating-pack + исходные модели.
Выход:
- VALIDATION_REPORT с критичными/средними/низкими проблемами.
Критерии приемки:
- Валидация проверяет: структуру, семантическую согласованность, операционную применимость.
- Каждая проблема содержит remediation-подсказку.
Риски:
- Ложные срабатывания и пропуск критичных ошибок.
Контроль:
- Порог критичности и ручная калибровка правил на пилоте.

Этап F. Pilot and Hardening
Цель:
- Подтвердить работоспособность V1 на реальных репозиториях.
Вход:
- V1 pipeline + 3-5 пилотных репозиториев разной структуры.
Выход:
- Отчет пилота, доработки, release-candidate V1.
Критерии приемки:
- Достижение целевых метрик (время, полнота, полезность первого чата).
- Закрыты блокирующие дефекты P0/P1.
Риски:
- Частные кейсы ломают общий pipeline.
Контроль:
- Тriage дефектов по приоритету; только критичное идет в релиз, остальное в post-MVP backlog.

Этап G. Release V1 Builder
Цель:
- Выпустить стабильный V1 как baseline.
Вход:
- RC-сборка + результаты пилота.
Выход:
- Release V1 + release notes + known limitations.
Критерии приемки:
- Документированы границы V1.
- Есть инструкция повторного запуска генерации.
- Зафиксирован backlog V2 Maintainer.

5) Контрольные точки согласования (Stage Gates)

Gate 1: Contract Gate
- Что согласуется: scope, артефакты, метрики V1.
- Решение: Go/No-Go к реализации.

Gate 2: Scanner Gate
- Что согласуется: качество факт-модели, confidence/unknowns.
- Решение: Go к опроснику и генератору.

Gate 3: Pack Quality Gate
- Что согласуется: полнота operating-pack и качество валидации.
- Решение: Go к пилоту.

Gate 4: Pilot Gate
- Что согласуется: достижение целевых метрик на пилоте.
- Решение: Go к релизу V1.

6) Артефакты, которые должны появиться по итогам V1

Обязательные продуктовые документы:
- V1 Scope Contract
- V1 Metrics & Acceptance Sheet
- Operating-Pack Template v1
- Validation Rulebook v1
- Pilot Report v1
- Release Notes v1
- V2 Backlog Seed (из реальных ограничений V1)

Обязательные пользовательские результаты:
- Сгенерированный operating-pack конкретного репозитория.
- Validation report по этому пакету.

7) Метрики и SLA для согласования V1

Core-метрики:
- Time-to-Pack: медиана до 30 минут.
- Mandatory Coverage: >=85% обязательных секций без ручного дописывания.
- First-Chat Utility: агент выполняет полезные первые действия с минимальными уточнениями.
- Critical Validation Failure Rate: <10% запусков.

Операционные SLA V1:
- Повторный запуск генерации после правок пользователя: доступен всегда.
- Ошибки сканирования не замалчиваются: всегда выводятся как unknown/error с пояснением.

8) Роли и зоны ответственности (для исполнения)

Owner/Product:
- Утверждает scope и stage-gates.
- Принимает решения по trade-offs между полнотой и скоростью.

Execution Team:
- Реализует pipeline по этапам A-G.
- Ведет реестр рисков и дефектов.

Pilot Users:
- Проходят сценарий запуска и дают обратную связь по полезности пакета.

9) Реестр рисков V1 и меры

Риск 1: Низкое качество сканирования
- Сигнал: много неверных фактов или пропусков.
- Мера: confidence + unknown-first стратегия + ручная проверка на Gate 2.

Риск 2: Плохая логика опросника
- Сигнал: пользователи бросают опрос или дают противоречия.
- Мера: лимит вопросов, приоритизация критичных неизвестных, conflict-check.

Риск 3: Слабая валидация
- Сигнал: в production-чате агент ошибается при «зеленом» отчете.
- Мера: добавить сценарные проверки в Gate 3/Pilot.

Риск 4: Сложность V1 разрастается
- Сигнал: в план попадают V2-фичи.
- Мера: жесткий scope freeze и backlog discipline.

10) План-график V1 (для согласования)

Фаза 1 (Недели 1-2):
- Contract Freeze + Scanner Baseline.
Exit:
- Gate 1 и Gate 2 (предварительный).

Фаза 2 (Недели 3-4):
- Questionnaire V1 + Generator V1.
Exit:
- Черновой полный operating-pack на тестовом репозитории.

Фаза 3 (Недели 5-6):
- Validator V1 + исправления качества.
Exit:
- Gate 3 (готовность к пилоту).

Фаза 4 (Недели 7-8):
- Pilot on 3-5 repos + hardening + релиз.
Exit:
- Gate 4 и Release V1.

11) Что нужно согласовать прямо сейчас

Критичные решения для старта:
- Подтверждение V1 scope (без функций V2/V3).
- Подтверждение фиксированного состава operating-pack (9 артефактов).
- Подтверждение stage-gates и метрик приемки.
- Подтверждение 3-5 пилотных репозиториев для финальной проверки.

Если эти 4 решения подтверждены, V1 можно запускать в исполнение без дополнительной декомпозиции.
